# Release v0.3.0 — Design

**Data:** 2026-07-14
**Branch:** `feat/v0.3.0-issues-3-4-5-6`
**Escopo:** GitHub issues #3, #4, #5, #6 numa única release.

## Contexto

O `LicenseManager` (pacote unlocked) adiciona uma camada analítica para determinar
utilização real de licenças Salesforce. A v0.2.x entregou o modelo per-purchase
(`LM_ProductLicensePurchaseCondition__c` + junction `LM_PersonaPurchaseAllocation__c`),
a automação de assignment (`LM_AssignmentGenerationBatch` / `LM_AssignmentAllocator`) e o
dashboard financeiro. A v0.3.0 fecha quatro lacunas conhecidas.

Os quatro issues **colidem nos mesmos arquivos wave**, então a ordem importa (ver
"Ordenação e dependências").

## Princípios de projeto herdados

- **Navegação hub-and-spoke:** `LM_ProductLicense__c` é a **única tab** do app; todos os
  demais objetos são navegação interna (related lists / links), nunca tab própria.
- **Prefixo `LM_`** em toda metadata; **traduções pt_BR** sob `objectTranslations/` mantidas
  em sincronia.
- **Package build + FLS:** testes que setam campos FLS-restritos devem usar `runAs` de um
  detentor da permset + `ProfileId = UserInfo.getProfileId()` (o runner do build não tem permset).

## Ordenação e dependências

| Ordem | Issue                        | Natureza                                    | Toca                                                   |
| ----- | ---------------------------- | ------------------------------------------- | ------------------------------------------------------ |
| 1     | **#5** UI da junction        | Declarativo isolado                         | layouts / flexipages / related lists / translations    |

resto.

## Seção 1 — #5: UI da junction de allocation

**Objetivo:** permitir que um admin crie/veja/ajuste as allocations Persona↔PurchaseCondition
pelo app, sem round-trip de data loader. Puramente declarativo — o modelo e a automação já
suportam.

- **Page layout** novo para `LM_PersonaPurchaseAllocation__c`: Persona (master-detail),
  PurchaseCondition (lookup), `Weight__c`, mais o contexto master-detail.
- **Flexipage** de record page, consistente com os outros 4 objetos.
- **Related lists** nos dois pais, editáveis in-context:
  - flexipage/layout do **`LM_ProductLicensePersona__c`** → allocations da persona;
  - flexipage/layout do **`LM_ProductLicensePurchaseCondition__c`** → quais personas/pesos
    alocam contra ele.
- **Sem tab** (hub-and-spoke) — edição só via related lists.
- **Traduções pt_BR** da junction (labels de campos / related list) em sincronia.
- **Reassessment leve:** confirmar que os lookups `PurchaseCondition__c` / `Persona__c` no
  `LM_ProductLicenseUserAssignment__c` já aparecem nas páginas (item do issue #5).

**Critério de aceite:** admin consegue criar e editar uma allocation a partir de ambos os pais,
sem usar data loader.

## Seção 2 — #4: Multi-org real

Hoje a separação B2B/B2C é **falsa**: campo texto manual `Org__c` em `LM_ProductLicense__c` +
filtros SAQL hardcoded + **seções hardcoded** no dashboard (14 refs B2B + 9 B2C no `.wdash`).

**Constraint de distribuição:** conexões Data Sync cross-org do CRM Analytics são específicas do
ambiente (credencial + org concreta) e **não são empacotáveis**. Não existe "recipe inativa com
conexões dummy" deployável — um input node que referencia connected object exige que a connection
resolva no deploy. Portanto o issue se divide em parte empacotada e extensão documentada.

### Parte empacotada (roda same-org)

- **Recipe/dataflow** (`License_Manager_Datasets_Preparation`) passa a carimbar uma dimensão
  **`SourceOrg`** nos datasets. Com uma org, é uma constante (label da org atual); a estrutura
  fica pronta para N ramos (carimbo por ramo de input).
- **Dashboard** (`.wdash`): substituir as seções/refs hardcoded B2B/B2C por **agrupamento
  dinâmico em `SourceOrg`**. Adicionar/remover org = dado, não edição de dashboard.
- **Drop de `Org__c`** em `LM_ProductLicense__c`: o campo, e todas as referências em
  `listViews/All`, flexipage, layout, e nas refs wave (1x dataflow, 3x recipe). A origem deixa de
  ser digitada à mão.
- **Sample data:** remover `Org__c` da coluna em `LM_ProductLicense__c.csv` e da query no
  `export.json` (`scripts/data/sfdmu/`).
- **Traduções pt_BR:** remover `Org__c` do objectTranslation.

### Extensão NÃO empacotada (admin faz no ambiente dele)

- **Runbook markdown** (`docs/`) — passo-a-passo: criar Data Sync connection para a org B (onde o
  app também está instalado), adicionar inputs dos `LM_` dela, carimbar `SourceOrg`, dar append nos
  datasets da org atual.
- **Recipe-sample de referência** no repo (`.wdpr` ilustrativo, **não** empacotado, aponta para
  orgs fictícias) mostrando o shape 2-orgs: 2 inputs → carimbo → append.

### Plano de validação hands-on (em conjunto)

1. Criar uma **segunda scratch org** ("org B").
2. Instalar o pacote LicenseManager nela + subir sample data (SFDMU).
3. Na **scratch original** (org A / hub), o **owner** cria a Data Sync connection apontando para
   a org B.
4. **Deploy** da sample recipe adaptada às connections reais (2 inputs → carimbo → append).
5. **Validação conjunta:** recipe roda, datasets têm ≥2 valores de `SourceOrg`, dashboard agrupa
   dinamicamente sem edição. O aprendizado realimenta o texto do runbook (passos reais, gotchas).

**Critério de aceite:** dashboard exibe org A + org B dinamicamente via `SourceOrg`, sem edição de
dashboard; nenhum resquício de `Org__c` no pacote nem no sample data.

## Seção 3 — #6: Tabela financeira agregada + facet chart→table

Na página financeira do `.wdash`, a `fin_table` hoje lista linhas cruas por-purchase, num grão
diferente dos charts (que agrupam `by ('License','Board')`) — as duas contam a mesma história em
grãos diferentes.

- **Agregar** a tabela a **uma linha por License** (breakdown de Board disponível), somando
  `Used / Contracted / IdleQty / UsedCost / IdleCost` — mesmo grão dos charts.
- **Facet chart→table:** seleção no chart (License, ou slice de Board) **filtra** a tabela via
  binding de `step`/selection.
- **Estado inicial:** sem seleção = **todas as licenças agregadas**; clicar filtra.
- **Ordem:** depois do #4, porque o #4 reestrutura essa mesma página (dinâmica em `SourceOrg`).

**Critério de aceite:** tabela no mesmo grão dos charts; clicar num bar filtra a tabela; sem
seleção, mostra todas as licenças agregadas.

## Seção 4 — #3: Aba de entitlements org-wide

**Realidade de plataforma** (pesquisa registrada): só **named** e **login/usage faturado**
atribuem a um purchase. API calls, storage e platform events são **pools org-wide** medidos contra
limites org-wide — dá para mostrar gauge "usado vs limite", mas **não** dividir por purchase/add-on
(storage nem API pública tem). A aba é honesta sobre isso.

- **LWC embarcado** no `.wdash` + **Apex controller fino** (`@AuraEnabled(cacheable=true)` sobre
  `OrgLimits.getAll()` — **sem callout, sem custom object, sem job agendado**). Expõe **todos os
  limites seguindo as seções da REST Limits API** (API requests, data/file storage, platform
  events, async apex, ...) como lista + gauges "usado vs máximo".
- **`TenantUsageEntitlement`** para login/usage **faturado** (`AmountUsed` vs
  `CurrentAmountAllowed`) — número comercial oficial, per-org. Fica no LWC ou step SOQL nativo
  (decisão de implementação).
- Tudo **rotulado como org-wide**, visualmente separado do modelo per-purchase (named).
- **`AssignmentType__c`** — normalizar para rotear famílias (Named / Login / Usage) no dashboard,
  na medida necessária para a navegação entre a aba named e a nova.

**Virada arquitetural (documentar no `CLAUDE.md`):**

- **Primeiro LWC + Apex controller** do pacote — o CLAUDE.md hoje afirma "no LWC/Aura, no
  triggers" e as pastas `lwc/`/`aura/`/`triggers/` estão vazias. Passa a ativar lint/jest de fato.
- Novo **Apex class access** na permset `LM_LicenseManager`; teste do controller respeitando o
  gotcha de FLS no package build.

**Critério de aceite:** aba nova exibe gauges org-wide de limites (via `OrgLimits`) + números
faturados de entitlement (via `TenantUsageEntitlement`), claramente separados do modelo
per-purchase; testes passam no package build.

## Fora de escopo (v0.3.0)

- Medição de **pico de login concorrente** (só estimável; não confiável).
- **Atribuição per-purchase** de API/storage/platform events (impossível na plataforma).
- **Snapshots históricos** de limites (exigiria custom object + job agendado — evitado de
  propósito; o LWC é leitura viva).
- Distribuição **empacotada** de conexão multi-org viva (não distribuível).
